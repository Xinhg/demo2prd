# proto-to-requirement Skill — 需求规格说明书

> 本文档定义了一个 Claude Code Skill，用于将交互原型（墨刀/Axure/Figma/HTML 导出包）逆向生成结构化需求文档。
> 遵循本规范开发的 Skill 应满足：全新子 Agent 仅依赖此 Skill 即可完成 85%+ 的需求文档生成。

---

## 1. 业务背景

### 1.1 为什么做这个 Skill

产品原型是需求落地的核心交付物，但原型本身无法直接驱动开发：

- 开发团队拿到原型后仍需人工梳理页面清单、交互映射、表单字段、业务规则
- 在 AI 辅助开发场景下，子 Agent 需要**结构化的需求文档**才能准确编码，而不是一个需要点击才能探索的链接
- 人工从原型写需求文档耗时且遗漏率高（平均遗漏 40%+ 的交互点）

**核心价值**：将"原型 → 开发"的链路缩短为"原型 → 需求文档 → 代码"，消除人工翻译环节。

### 1.2 解决的业务问题

| 痛点 | 解决方案 |
|------|---------|
| 原型交互需逐页点击探索 | 自动提取全量交互映射（1,164 个交互 → 88 个目标页面的完整矩阵） |
| 表单字段需人工逐一记录 | 自动提取表单组件清单并配对 label 文本 |
| 业务规则分散在批注中难以汇总 | 专门扫描批注组件，提取纯文本规则 |
| 需求文档完整度无法量化 | 每次输出附带完整性自评，标注"事实/推断/未知"三级 |
| 视觉信息利用率低 | 支持视觉模式，渲染截图 → 视觉模型描述 → 合并为需求 |

## 2. 用户角色

| 角色 | 职责 | 使用场景 |
|------|------|---------|
| **产品经理** | 上传原型包，验证输出文档准确性 | "帮我从这份墨刀导出包生成需求文档" |
| **技术负责人** | 用输出文档驱动子 Agent 编码 | "生成文档后，开个子 Agent 按文档实现" |
| **QA 测试** | 用交互映射表设计测试用例 | "这个页面的所有可点击元素和目标是什么" |

## 3. 主流程

### 3.1 整体架构

```
原型包 (zip/目录)
    │
    ▼
Phase 1: 格式探针 (format-probe)
    │  识别工具类型 + 定位数据文件 + 检测编码/压缩
    ▼
Phase 2: 数据解包 (unpack)
    │  解压 → 解码 → 结构化 JSON
    ▼
Phase 3: 并行提取 (3 条线)
    ├── 页面清单 + 组件树
    ├── 交互映射 (重点)
    └── 文本内容 + 批注
    │
    ▼
Phase 4: (可选) 视觉增强
    │  渲染页面 → 截图 → 视觉模型分析
    │  → 表单字段配对、下拉选项识别、表格列头提取
    │  → 状态差异对比、设计语义理解
    ▼
Phase 5: 双通道合并 + 文档渲染
    │  数据通道 + 视觉通道 → 交叉验证 → Markdown 输出
    ▼
Phase 6: 完整性自评
    │  按模块评分 → 标注未覆盖项
    ▼
输出: 需求规格说明书 + JSON 结构化数据
```

### 3.2 Phase 1: 格式探针

**输入**：目录路径

**步骤**：

1. 扫描目录结构，匹配已知特征模式：

| 特征 | 判定为 | 数据文件定位 |
|------|--------|------------|
| `mb-proto2/` + `extra/data.*.js` | 墨刀 Mockitt | `extra/data.1.js`（主数据）<br>`extra/data.0.js`（配置元数据） |
| `resources/` + `data/` 含 `.rp` 文件 | Axure RP | `data/document.js` |
| `*.fig` 文件 | Figma export | `.fig` 直接解析 |
| 含 `app.js` + 路由配置 + `pages/` | Sketch/代码导出 | 遍历 `pages/` 下 HTML |
| `index.html` + `static/` + `config.json` | 通用前端框架 | 读取 `config.json` 路由表 + 页面入口 |

2. 读取数据文件的前 500 字节，检测编码/压缩：
   - 以 `H4sIAAAA` 开头 → gzip + base64
   - 以 `window["xxx"]` 开头 → JS 变量赋值，内含 JSON
   - 以 `{` 或 `[` 开头 → 原始 JSON

3. 输出：`{tool_type, data_files: [...], encoding, compression, entry_file}`

**异常处理**：
- 无法识别工具类型 → 回退到通用模式：遍历所有 >10KB 的 JS/JSON 文件，尝试解析
- 多个可能的数据文件 → 按文件大小降序，最大的通常包含主数据

### 3.3 Phase 2: 数据解包

**输入**：Phase 1 输出的 `data_files`

**步骤**（以墨刀为例）：

1. 读取主数据文件的 JS 变量赋值，提取其中的 JSON/编码字符串
2. 检测压缩格式并解压（gzip → zlib → brotli → 尝试原始）
3. 解析顶层结构——通常为行分隔格式（`行ID\t{JSON属性}`）或纯 JSON
4. 构建组件数据库 `{component_id: {properties}}`：

```
关键发现：组件 ID 可能带有 2 字符后缀（如 " ." 或 " ,"）
          这些后缀在跨引用查找时需要统一 strip
```

5. 输出：`{pages: [...], components: [...], raw_components_count, pages_count}`

**异常处理**：
- 解压失败 → 尝试 zlib raw（无 header）、brotli、或直接 base64 解码
- JSON 解析失败 → 检查是否是单行 JSON vs 多行 JSON vs NDJSON
- 组件 ID 查找失败 → 自动检测 ID 后缀模式，统一 strip 后重试

### 3.4 Phase 3: 并行提取

三条线同时进行，互不依赖。

#### 3.4.1 页面清单提取

**输入**：`components` 数据集

**方法**：
1. 筛选有特定前缀的容器组件（墨刀为 `rbp`）
2. 提取其 `N`（名称）字段作为页面名
3. 去重（同一页面可能有多个状态变体，名称相同但 ID 不同）

**输出**：`[{page_id, page_name, is_state_variant}]`

#### 3.4.2 交互映射提取（核心路径）

**输入**：`components` 数据集

**方法**：

```
1. 遍历所有组件，筛选含 "I" 字段的
2. 对于每个交互条目 I[i]:
   - I[i][0]: 交互类型码
   - I[i][1]: 交互目标 ID
   - I[i][4]: 上下文数组
     - I[i][4][1]: 目标页面的 rbp ID → 通过页面名查找表解析为人类可读的名称
     - I[i][4][2]: 目标状态实例 ID
3. 构建三元组: (源组件名, 交互类型, 目标页面名)
4. 按目标页面分组
```

**关键启发式规则**：
- 如果直接的目标 ID 查找失败，检查上下文数组中的冗余引用
- 交互类型码通常是：1=页面跳转, 2=返回/关闭, 3=弹窗, 10=状态切换——但不同工具编码不同，需从样本确认
- 同一个源组件可能有多个交互（主操作跳转 + 备用操作打开弹窗）

**输出**：`[(source_component, interaction_type, target_page), ...]` + `{target_page: [entry_points]}`

#### 3.4.3 文本内容提取

**输入**：`components` 数据集

**方法**：
1. 遍历所有组件的以下字段：
   - `N`（组件名/标签）
   - `b/#000000`（按钮文本/可见文字内容）
   - `rtS`（富文本样式中的文本内容）

2. 特殊处理：筛选 `T` 类型为批注/标注的组件（墨刀中为 `)-)` 或 `i` 类型），提取其中的中文文本作为业务规则

3. 去重并按页面分组

**输出**：
- `[{page, component_name, text_content}]`
- `[{page, annotation_text}]`（业务规则候选集）

### 3.5 Phase 4: 视觉增强（可选，需视觉模型）

**前提**：存在可渲染的 `index.html` 入口，或原型在线链接。

**步骤**：

1. **页面渲染与截图**：
   - 在浏览器中打开原型
   - 遍历所有页面状态，对每个页面截图
   - 命名规则：`{页面ID}_{状态序号}.png`

2. **表单字段配对**（视觉模型任务）：
   ```
   提示词：在这张截图中，找出所有表单字段。
   对每个字段，输出：[label文本, 控件类型, placeholder文本, 是否必填]
   控件类型枚举：text_input, textarea, select, radio, checkbox, date_picker, time_picker, file_upload, switch
   ```

3. **下拉选项提取**（视觉模型任务）：
   ```
   提示词：在这个下拉选择器的展开截图中，列出所有选项值。每行一个。
   ```

4. **表格结构提取**（视觉模型任务）：
   ```
   提示词：识别这张截图中的表格。
   输出列头序列和一行示例数据。[列1: 值1, 列2: 值2, ...]
   ```

5. **状态差异分析**（视觉模型任务）：
   ```
   提示词：对比这两张同一页面的截图。
   描述可见差异：新增/消失/变色/文字变化的元素。
   ```

6. **设计语义理解**（视觉模型任务）：
   ```
   提示词：描述页面的整体布局结构。
   识别：导航类型、内容区域分组、按钮层级（主/次/文字链接）、
   表单分区、空状态占位、错误状态提示。
   ```

**输出**：`visual_observations.json`（与数据通道输出的并集可交叉验证）

### 3.6 Phase 5: 双通道合并

**输入**：数据通道输出 + 视觉通道输出

**合并策略**：

| 场景 | 策略 |
|------|------|
| 数据有、视觉也有 → 交叉验证，不一致标记为需人工确认 |
| 数据有、视觉无 → 保留，标注"(数据提取)" |
| 视觉有、数据无 → 采用，标注"(视觉提取)" |
| 两者都无 → 标注"未提供相关信息" |

**文档渲染**：按标准需求文档模板组织为 10 个章节（见第 8 节）。

### 3.7 Phase 6: 完整性自评

**评分维度**：

| 维度 | 权重 | 评分依据 |
|------|:---:|---------|
| 页面覆盖率 | 20% | 已识别页面 / 交互涉及的页面 |
| 交互映射率 | 25% | 已解析交互 / 总交互数 |
| 表单字段率 | 20% | 已知字段 / 表单总数 |
| 表格列名率 | 10% | 已知列名 / 表格总数 |
| 业务规则提取 | 10% | 从批注中提取的规则数 |
| 下拉选项率 | 5% | 已知选项 / 下拉总数 |
| 状态差异覆盖率 | 5% | 已描述的状态差异 / 状态变体总数 |
| 不确定性占比 | 5% | 标注为"事实"的条目占比 |

**可实现度映射**：
- 90%+：子 Agent 可独立完成全部代码
- 75-89%：子 Agent 可完成大部分，少量需人工补充
- 60-74%：子 Agent 可完成骨架和主要流程
- <60%：文档不足以驱动开发，需人工介入

## 4. 异常流程

| 场景 | 处理方式 |
|------|---------|
| 无法识别原型工具 | 回退到通用模式：遍历所有 >10KB 的 JS/JSON，尝试通用结构解析 |
| 数据解压失败 | 依次尝试 gzip → zlib → brotli → raw base64 → 原始文本 |
| 组件 ID 查找失败 | 自动检测 ID 后缀模式（如 " ."、" ,"），统一 strip 后重试；记录失败日志 |
| 交互目标无法解析 | 保留原始 ID，标注"未解析"，不丢弃数据 |
| 页面名大量重复 | 按页面 ID 前缀匹配去重，同名视为同一页面的不同状态变体 |
| 视觉模型不可用 | 降级为纯数据模式，表单字段率从 85% 降至 15%，在下一次评评估中标注 |
| 原型无渲染入口 | 跳过视觉增强阶段，不阻塞主流程 |
| 数据文件超过内存限制 | 分块读取，流式解压；超大文件仅提取关键字段（I/N/T） |
| 编码格式未知 | 统计字符集大小 → 尝试常见 base-N 解码 → 失败则保留编码值 |

## 5. 业务规则 / 核心原则

### 5.1 格式优先（Format-First）

> 花 20% 的时间搞清楚数据格式，能省下 80% 的返工。

先回答三个问题再动手提取：哪个工具？数据在哪个文件？什么编码？

### 5.2 关系追踪（Trace Relationships）

> 原型 = 页面 × 组件 × 交互。价值在交互边上。

永远优先提取交互映射（谁→到哪里），再填充页面内的静态内容。交互映射是需求文档的骨架。

### 5.3 语义深挖（Semantic Over Encoding）

> 按钮上的文字比按钮的像素位置重要 100 倍。

编码值（坐标、颜色、圆角）对需求文档价值为零。跳过它们，深挖文本和交互 ID。

判断标准：
- **必须提取**：组件名称、文本内容、交互目标、组件类型、批注
- **选择性提取**：表格列宽/行高（可用于布局推断）
- **果断跳过**：像素坐标、颜色值、阴影参数、动画曲线

### 5.4 批注即规则（Annotations Are Business Rules）

> 设计师留下的旁白，比 UI 元素更有业务价值。

专门扫描批注/标注类组件，提取纯文本。这些文本通常是产品经理和设计师沟通后留下的决定性信息。

### 5.5 冗余引用利用（Leverage Redundant References）

> 数据里通常有"抄近路"的冗余引用，利用它们避免二次查找。

例如：交互条目中交互目标 ID 可能不可直接解析，但同一条目的上下文数组里往往直接引用了目标页面的 ID——这是在数据序列化时留下的冗余，恰好可以跳过中间查找步骤。

### 5.6 三级标注（Three-Tier Uncertainty）

> 文档使用者必须能区分：事实、推断、未知。

```
✅ 事实：从数据中直接提取的，不经过任何推理
⚠️ 推断：基于上下文模式推测的，标注"(推断)"
❌ 未知：数据中不存在，标注"未提供相关信息"
```

这不是谦虚，是工程必需品。如果子 Agent 把推断当事实，写出来的代码就有 bug。

### 5.7 迭代加深（Iterative Deepening）

> 广度优先 → 深度优先 → 按需 drill down。

- 第一轮：页面清单 + 交互计数（5 分钟）
- 第二轮：交互映射 + 文本提取（10 分钟）
- 第三轮：按模块 drill down 表单字段和表格列（按需）

### 5.8 完整性自评（Completeness Self-Assessment）

> 文档生成完毕后，必须回答：一个全新上下文的子 Agent 能写出多少代码？

如果不做这个评估，文档质量就无法量化——你会以为交付了 90% 的完整文档，实际上可能只有 35%。

### 5.9 双通道验证（Dual-Channel Verification，视觉模式专属）

> 数据和视觉是两个独立信息源，交叉验证能发现单通道盲区。

当视觉发现和数据不一致时：标记冲突 → 增加置信度较低方的标注 → 不强制选一边。

## 6. 数据规则

### 6.1 输入格式

| 输入 | 格式 | 说明 |
|------|------|------|
| 原型目录 | 本地路径 | 完整的原型导出目录 |
| 或原型包 | .zip / .tar.gz | 先解压到临时目录 |
| 或在线链接 | URL | 墨刀/Figma 在线预览链接（需视觉模式） |
| 可选参数 | JSON | `{output_path, enable_visual, visual_model, target_language}` |

### 6.2 输出格式

| 输出文件 | 格式 | 说明 |
|---------|------|------|
| `需求规格说明书.md` | Markdown | 10 章节格式化文档 |
| `structured-data.json` | JSON | 机器可读的结构化数据 |
| `completeness-report.json` | JSON | 完整性评分和缺失项清单 |

### 6.3 数据流 schema

```json
{

    "tool_type": "mockitt|axure|figma|sketch|generic",

    "project_name": "string",

  },
  "pages": [
    {
      "page_id": "string",
      "page_name": "string",
```

### 6.4 辅助脚本清单

| 脚本 | 输入 | 输出 | Phase |
|------|------|------|:----:|
| `format-probe.py` | 目录路径 | `{tool_type, data_files, encoding}` | 1 |
| `unpack-data.py` | data_files | `{pages, components}` JSON | 2 |
| `extract-pages.py` | components | `[{page_id, name}]` | 3.4.1 |
| `map-interactions.py` | components + page_names | `[(src, type, target)]` | 3.4.2 |
| `extract-text.py` | components | `[{text, annotations}]` | 3.4.3 |
| `visual-analyze.py` | screenshots_dir + visual_model | `visual_observations.json` | 4 |
| `merge-channels.py` | data_channel + visual_channel | 合并后结构化数据 | 5 |
| `render-doc.py` | structured_data | 需求文档 .md | 5 |
| `assess-completeness.py` | structured_data | 完整性报告 | 6 |

### 6.5 校验规则

| 校验项 | 规则 |
|--------|------|
| 交互源必须在组件清单中 | 每个交互的 source_component 必须存在于某页面的组件列表，否则标记为 orphan interaction |
| 目标页面必须存在 | 每个交互的 target_page 必须在页面清单中，否则标记为 unresolved target |
| 文本内容不能全部为空 | 如果提取的文本条目中 <10% 包含有效中/英文内容，警告可能解析错误 |
| 页面不能全部同名 | 如果去重后页面名 <3 个，警告分类逻辑可能错误 |

## 8. 验收标准

### 8.1 功能验收

- [ ] 支持识别至少 3 种原型工具（墨刀/Axure/Figma）

## 10. 附录

### 10.1 墨刀数据格式参考

以下信息有助于后续适配和维护：

```
数据层次:
  @@R *        — 空引用表
  @@M 1        — 元数据条目
  B@main 1     — 主页面容器
  B@trash      — 回收站
  B@ref-muban  — 模板引用
  rbp*         — 页面/容器定义 (217 个)
  rc*          — 组件实例引用 (319 个)
  V*           — 实际 UI 组件 (5,172 个)

组件关键字段:
  N: 名称
  T: 类型 () =container *=root M=div P=span x=text R=frame a=select c=input b=textarea s=radio w=table y=icon S=image Q=circle ]=button i=annotation )-=dot-annotation
  I: 交互定义数组 [[type, target_id, "", "(", [context_data], flags, ...], ...]
  B: 子容器引用键
  rRBPK: 子组件列表引用 ID
  rCK: 第一个子组件引用 ID
  b/#RRGGBB: 文本内容 ["flags", "text", [styles], [effects]]
  xy: 编码坐标 (base-86)
  wh: 编码尺寸 (base-86)

交互类型码:
  1: 页面导航    2: 返回/关闭    3: 弹窗    10: 状态切换

交互上下文数组 I[i][4]:
  [0]: 触发条件标志
  [1]: 目标页面 rbp ID (可直接解析为页面名)
  [2]: 目标状态实例 rc ID
  [3-4]: 视觉标志 ("h")

ID 后缀: " ." 或 " ," 
  组件原始 ID 带有 2 字符后缀，跨引用查找必须统一 strip
```

### 10.2 已知限制

1. **API 契约不可获取**：原型不包含后端信息，API schema 必须人工补充
2. **校验规则不完整**：原型只能展示校验失败状态，无法获取完整校验规则
3. **计算逻辑不透明**：数值计算（如保费统计）的后端逻辑不可从原型获取
4. **权限矩阵不可见**：原型只展示"有权限管理页面"，不展示具体权限分配
5. **工具演进风险**：墨刀可能更新数据格式，需要定期验证解析逻辑
